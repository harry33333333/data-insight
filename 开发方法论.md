# AI辅助软件开发操作说明（完整版）

### 第一部分：核心公理与定理体系

以下公理和定理是整个操作方案的理论基石，所有实践步骤均由此推导。

#### 重构后的五大核心公理

| 公理编号 | 公理名称 | 核心表述 |
| --- | --- | --- |
| **公理一** | **输入上限原理** | AI输出的质量上限，由输入给它的意图清晰度、上下文完备性和约束精确性共同决定。模糊的输入不可能稳定地产出精确的软件。 |
| **公理二** | **责任不可让渡** | 软件开发者对最终产品承担完全且不可转移的责任。任何AI生成的代码，都必须可被人类开发者完全解释、维护和调试。 |
| **公理三** | **原子组合必然性** | 可靠的复杂软件系统，只能由独立可验证的原子单元确定性地组合而成，不存在从混沌需求直接生成可靠系统的一次性捷径。 |
| **公理四** | **输出可错性** | AI的每个输出，在未经验证之前，都仅仅是基于统计模式的假设，必然包含非零概率的错误、幻觉或安全缺陷，不具备天然正确性。 |
| **公理五** | **认知循环守恒** | 软件开发的本质是“假设→验证→修正”的认知循环。AI可压缩单次循环的时间成本，但不可消除循环本身，且循环中的决策权与反思必须保留在人类。 |

#### 从公理推导出的所有定理

**从公理一（输入上限）推导**

*   **定理1.1（废进废出定理）**：若输入信息中存在歧义、矛盾或信息缺失，则AI输出中必然存在对应的不确定性或错误。垃圾信息进，垃圾代码出。
    
*   **定理1.2（意图完备性收益递减定理）**：意图清晰度对输出质量的提升存在边际效用递减。一旦超过AI能力边界（如训练数据覆盖范围、推理深度极限），追加意图不再提高输出质量。
    

**从公理二（责任不可让渡）推导**

*   **定理2.1（理解必然性定理）**：开发者必须能够独立解释和维护AI生成的每一行关键代码。若某段代码无法被理解，则其等同于未被验证的第三方黑箱，违反公理二。
    
*   **定理2.2（签名即责任定理）**：无论代码来源如何，合入主分支的代码被视同开发者本人所写，承担同等责任。禁止以“这是AI写的”为由推卸任何事故责任。
    

**从公理三（原子组合必然性）推导**

*   **定理3.1（一次性生成不可靠定理）**：让AI一次性生成超过原子单元粒度的复杂功能模块，其正确性概率随代码长度指数下降，无法通过工程可靠性要求。
    
*   **定理3.2（接口契约锁定定理）**：在AI辅助开发中，必须先由人类设计并锁定模块间的接口契约（函数签名、数据格式），再让AI填充实现。先定边界，后填内容。
    
*   **定理3.3（单元测试前置定理）**：每个AI生成的原子模块，必须先有测试用例定义其预期行为，再生成实现代码。没有测试的AI输出，不具备可验证性。
    

**从公理四（输出可错性）推导**

*   **定理4.1（验证不可替代定理）**：AI输出不存在可以跳过验证的捷径。任何声称“AI生成的代码无需测试”的实践，必然累积技术债务或引入缺陷。
    
*   **定理4.2（信任与风险反比定理）**：对AI输出的信任程度，必须与代码失效后的风险程度成反比。风险越高的模块（安全、支付、数据），对AI输出的初始信任应越低，验证应越严格。
    
*   **定理4.3（幻觉湮灭定理）**：AI幻觉只能通过外部事实源（官方文档、运行环境、类型检查器、测试执行）进行交叉验证来湮灭，无法通过同一模型的自我反思来根除。
    

**从公理五（认知循环守恒）推导**

*   **定理5.1（循环不可跨越定理）**：软件开发的任何增量，都必须经历至少一次完整的“假设→验证→修正”循环。AI可让单次循环缩短至秒级，但无法让循环次数降为零。
    
*   **定理5.2（决策权定理）**：在认知循环中，关于“是否接受输出”、“何时停止迭代”、“方案如何取舍”的决策权不可委托给AI。AI是回音壁，人类是决策者。
    
*   **定理5.3（反思不可自动化定理）**：开发过程中对方案优劣的权衡、对架构取舍的判断、对技术债务的感知，必须由人类完成。AI可提供分析素材，但反思本身无法外包。
    

**公理-定理体系一览**

| 公理 | 核心表述 | 派生定理数 |
| --- | --- | --- |
| 公理一：输入上限 | 输出质量上限由输入决定 | 2条 |
| 公理二：责任不可让渡 | 开发者承担完全责任 | 2条 |
| 公理三：原子组合必然性 | 可靠系统只能由原子单元组合 | 3条 |
| 公理四：输出可错性 | AI输出包含非零错误概率 | 3条 |
| 公理五：认知循环守恒 | 开发循环不可消除，决策在人 | 3条 |
| **总计** |  | **5条公理，13条定理** |

### 第二部分：五阶段实战操作流程

本流程将上述理论转化为具体步骤，核心原则是：**人类主导设计、决策与验收，AI负责在严密约束下执行实现。**

#### 整体流程：五阶段闭环

需求原子化 → 接口契约锁定 → TDD循环生成 → 批判性验证 → 所有权确认

---

#### 阶段一：需求原子化（落地公理一、三 / 定理1.1、3.1）

**目标**：把模糊需求拆成AI可消化的独立原子任务。

**操作步骤**：

1.  **需求拆解清单**：
    
    *   用自然语言写出功能，然后强制拆分为“一个函数/一个类只做一件事”的粒度。
        
    *   每个原子任务必须满足：单一职责、无歧义输入输出、不依赖其他未完成模块。
        
2.  **编写“任务委托书”模板**：

3.  **检查点**：
    
    *   如果一个任务描述超过一屏，说明粒度太大，继续拆分。
        
    *   如果任务存在“和/或”等并列词，拆成两个任务。
        

---

#### 阶段二：接口契约锁定（落地公理三 / 定理3.2）

**目标**：先定边界，后填内容。人类设计接口，AI填充实现。

**操作步骤**：

1.  **先写接口定义**：
    
    *   在让AI生成实现前，人类先写好函数签名、类型定义、数据格式。
        
2.  **将接口写入项目规则文件**：
    
    *   在Cursor Rules / .cursorrules / 自定义指令中，声明：“所有生成代码必须实现已有接口，不得修改接口签名。”
        
    *   或每次对话时，将接口定义作为固定前缀粘贴给AI。
        
3.  **锁定数据契约**：
    
    *   若涉及数据库，先由人类设计Schema和迁移脚本，AI只填充CRUD实现。
        
    *   若涉及API，先定义OpenAPI/JSON Schema，AI只实现处理函数。
        
4.  **检查点**：
    
    *   接口是否有版本号或变更记录？
        
    *   接口的异常路径是否已定义（不是只定义Happy Path）？
        

---

#### 阶段三：TDD循环生成（落地公理四、五 / 定理3.3、5.1）

**目标**：用测试驱动AI生成代码，一次一个原子模块。

**操作步骤（严格按此顺序）**：

1.  **先写测试用例（由人类或AI辅助写）**：
    
    ```typescript
    describe('AuthService.login', () => {
      it('正确密码应返回token', async () => { ... })
      it('错误密码应返回error', async () => { ... })
      it('不存在用户应返回error，且不泄露是否存在', async () => { ... })
    })
    
    ```
    
2.  **将测试发给AI，生成实现**：
    
    *   提示词：“以下是测试用例，请编写满足这些测试的AuthService.login实现。不要修改测试代码。”
        
    *   运行测试 → 红灯（测试失败）是预期状态。
        
3.  **AI生成实现 → 运行测试 → 绿灯**：
    
    *   测试不通过？把报错信息发给AI修复，最多循环3次。
        
    *   3次后仍未通过，人工介入分析，不再盲目重试。
        
4.  **重构（仅在有测试保护下）**：
    
    *   让AI优化代码结构、性能，要求“所有现有测试必须继续保持通过”。
        
    *   重构后立即运行全量测试。
        
5.  **循环节奏**：
    
    *   一个原子任务 = 一个TDD循环。
        
    *   一个循环理想耗时：5-15分钟。
        
    *   一个循环产出：一个经过测试验证的原子模块。
        

---

#### 阶段四：批判性验证（落地公理四 / 定理4.1、4.2、4.3）

**目标**：每一行AI代码都经过多层防御验证。

**操作步骤：四层验证**：

1.  **第一层：自动化静态检查（零成本，必须过）**：
    
    *   TypeScript/ESLint/Lint：无错误。
        
    *   类型检查：严格模式，无any逃逸。
        
    *   AI生成的代码合入前，必须通过CI管线的静态检查。
        
2.  **第二层：自动化动态检查（单元测试+集成测试）**：
    
    *   每个原子模块必须有对应测试，覆盖率不做硬性指标，但关键路径和异常路径必须覆盖。
        
    *   任何AI修改现有代码，必须跑全量回归测试。
        
3.  **第三层：人工代码审查（聚焦高风险区域）**：
    
    *   审查清单（对照定理4.2的“信任与风险反比”）：
        
        | 风险等级 | 代码类型 | 审查要求 |
        | --- | --- | --- |
        | 高 | 认证、支付、权限、数据迁移 | 逐行审查，必须完全理解每行逻辑 |
        | 中 | 业务逻辑、数据处理 | 理解整体流程，抽查关键分支 |
        | 低 | UI模板、格式化、日志 | 肉眼扫读，确认无敏感信息泄露 |
        
    *   审查必问四个问题：
        
        *   这段代码有对应的测试吗？测试覆盖了异常路径吗？
            
        *   我能否在不求助AI的情况下解释这段代码的行为？
            
        *   是否存在硬编码、魔法数字、或暴露敏感信息？
            
        *   错误处理是否恰当？会不会静默吞掉异常？
            
4.  **第四层：外部事实交叉验证（落地定理4.3）**：
    
    *   AI引用的第三方库版本、API用法，必须去官方文档验证一次（至少确认存在该函数/版本）。
        
    *   对安全相关代码（加密、认证），必须使用已知安全的库函数，禁止AI自创加密逻辑。
        
    *   对并发/事务代码，必须有明确的测试模拟竞态条件。
        

---

#### 阶段五：所有权确认（落地公理二 / 定理2.1、2.2）

**目标**：确保代码库的每一行都有明确的人类责任人。

**操作步骤**：

1.  **“橡皮鸭解释”规则**：
    
    *   每天至少随机抽取一段AI生成的代码，用自己的话向同事（或橡皮鸭）解释其逻辑。
        
    *   解释不通 → 标记为技术债务，当天必须弄清或重构到可解释状态。
        
2.  **Git提交规范**：
    
    *   禁止提交消息写“AI generated code”。
        
    *   必须写清楚“实现XX功能，采用YY方案，因为ZZ原因”。
        
    *   署名是你的名字，责任是你的。
        
3.  **维护文档同步**：
    
    *   每完成一个模块，更新架构决策记录（ADR），记录：为什么选择这个方案、AI参与了哪些部分、已知限制是什么。
        
    *   ADR示例：
        
4.  **“杀死黑箱”机制**：
    
    *   代码库中不允许存在“它能工作但我不知道为什么”的模块。
        
    *   技术债看板中设立“AI黑箱消除”类别，定期排期重构。

**核心心法：一句话操作指南**

把AI当成一个速度极快但需要你承担全部责任的初级程序员——你怎么带新人，就怎么带AI。 你写任务书（阶段一），你定接口（阶段二），你写测试（阶段三），你验收（阶段四），你签责任状（阶段五）。AI只填充“实现”这一个环节，但整个流程的节奏和质量由你掌控。

### 第三部分：多Agent分工协作指南

在多Agent协作时，必须遵循 **“契约驱动流水线”** 模式：所有Agent不直接对话，只通过人类定义的契约（接口、测试）和人类指令协作。

**必须牢记一条铁律**：

Agent只是你的“专业助手”，它们分担执行，你保留决策、验收和最终责任。 这与公理二（责任不可让渡）完全一致。

#### 一、Agent角色定义（始终保留人类决策权）

| 角色 | 职责 | 不可做的事 | 对应公理/定理 |
| --- | --- | --- | --- |
| **人类主工程师** | 定义目标、拆分任务、设计接口、最终验收、承担责任 | — | 公理二、定理2.2、定理5.2 |
| **规划Agent** | 辅助需求原子化，生成任务委托书草稿 | 不得自行决定最终架构 | 公理一、定理1.1 |
| **实现Agent** | 在锁定接口和测试下生成代码 | 不得修改接口、不得逃避测试 | 公理三、定理3.2、3.3 |
| **测试Agent** | 编写测试用例、执行测试、报告结果 | 不得生成无断言的假测试 | 公理四、定理4.1 |
| **审查Agent** | 静态分析、模式检查、常见漏洞扫描 | 不得代替人类进行最终合入审批 | 公理四、定理4.2 |
| **文档Agent** | 生成接口文档、更新ADR草稿 | 不得虚构尚未实现的功能 | 定理2.1 |

#### 二、Agent协作的核心法则：契约驱动流水线

所有Agent之间不直接对话，只通过人类定义的“契约”间接协作。 这保证了每个Agent的输出都是可隔离验证的原子单元（公理三）。

```plaintext
人类定义接口契约 + 测试用例
          |
          v
    ┌─────┴─────┐
    │ 实现Agent  │ ← 只消费：接口+测试
    └─────┬─────┘
          | 生成实现代码
          v
    ┌─────┴─────┐
    │ 测试Agent  │ ← 消费：实现代码 + 原有测试
    └─────┬─────┘
          | 报告：通过/失败 + 日志
          v
    ┌─────┴─────┐
    │ 审查Agent  │ ← 消费：实现代码 + 项目规范
    └─────┬─────┘
          | 报告：告警列表
          v
       人类决策

```

#### 三、五阶段流程的Agent植入步骤

**阶段一：需求原子化（规划Agent辅助）**

*   人类：用自然语言描述顶层功能。
    
*   规划Agent：接收描述，根据模板生成原子任务拆解建议，输出任务委托书列表。
    
*   人类：逐一审核、修正、确认粒度（一个函数/类），最终锁定任务清单。
    
*   **禁止**：让规划Agent直接生成任务依赖关系图并以此调度执行——依赖关系必须由人类审视，避免循环依赖或过度拆分。
    

**阶段二：接口契约锁定（人类主导，Agent辅助生成草案）**

*   人类：为每个原子任务定义接口签名（类型、函数名、数据格式）。
    
*   实现Agent：可请求其生成接口定义的备选草案（例如TypeScript interface），但人类必须逐字段确认语义。
    
*   文档Agent：基于接口自动生成OpenAPI片段或JSDoc草稿，人类审核后锁定。
    
*   规则注入：将所有接口定义写入项目Agent共享规则（如cursorrules），确保每个Agent都能看到“契约上下文”。
    

**阶段三：TDD循环生成（多Agent串行协作）**

这是Agent分工的核心环节。按以下严格顺序：

1.  **测试Agent先行**：
    
    *   提示词：“根据此接口契约和业务规则，生成全面的测试用例，覆盖正常路径、异常路径和边界条件。”
        
    *   人类审核：检查测试是否真的验证了业务规则，有无遗漏的断言，有无只调用不验证的假测试。
        
2.  **实现Agent编码**：
    
    *   提示词：“以下是已锁定的测试代码和接口。请编写满足所有测试的实现，不要修改测试和接口。若无法实现，请明确指出冲突点并停止，不要猜测。”
        
    *   **关键**：给实现Agent的上下文只包含测试和接口，不额外透露实现思路，以测试为唯一正确性标准。
        
3.  **测试Agent验证**：
    
    *   自动化管道：收到实现代码后，立即触发测试Agent执行。
        
    *   若红灯：将失败日志+实现代码一起提交给同一个实现Agent修复，最多允许3次尝试。超过3次，人类介入分析原因，不得无限循环（防止定理4.1边际递减）。
        
4.  **重构（可选）**：
    
    *   测试全部绿灯后，可引入重构Agent：“优化代码结构，要求所有现有测试继续保持通过。”
        
    *   重构后必须再次触发测试Agent全量验证。
        

**阶段四：批判性验证（多Agent并行审查）**

在合入前，启动三道自动化防线，最后人类终审：

| 防线 | Agent类型 | 动作 | 输出 |
| --- | --- | --- | --- |
| 1 | 审查Agent（静态） | 运行lint、类型检查、安全模式扫描（硬编码密钥、SQL注入等） | 阻塞性错误必须清零 |
| 2 | 测试Agent（动态） | 运行原子模块测试+回归测试套件 | 全绿通过 |
| 3 | 审查Agent（依赖审计） | 检查新引入的第三方库调用是否与官方文档一致（定理4.3） | 告警列表供人类复查 |

*   人类终审聚焦清单（来自定理4.2）：
    
    *   高风险的认证、支付、数据迁移代码：逐行阅读。
        
    *   所有Agent告警：逐一确认或驳回。
        
    *   橡皮鸭测试：随机抽取一段代码，当场解释。
        

**阶段五：所有权确认（Agent辅助文档，人类签字）**

*   文档Agent：根据本次变更生成ADR草稿和更新API文档。
    
*   人类：修正文档，确保如实记录AI参与部分和已知限制。
    
*   Git提交：人类亲自执行，提交信息写明方案原因，签名。
    
*   **禁止**：使用Agent自动生成提交信息并直接推送，那会让你失去最后一道所有权痕迹。
    

#### 四、Agent分工的三大安全准则

1.  **自主权阶梯原则**：
    
    *   信任从零开始：Agent输出必须默认不信任，经过上述流水线后才可逐步建立信任。
        
    *   对于简单、无副作用的工具函数，可给予实现Agent更高自主性，但仍需测试验证。
        
2.  **上下文隔离原则**：
    
    *   不同Agent之间不共享“思考过程”，只交换契约输出（接口、测试结果、日志）。这避免了上下文污染导致的虚假一致。
        
    *   例如，不要让审查Agent看到实现Agent的“解题思路”，只让它看最终代码。
        
3.  **紧急熔断原则**：
    
    *   设定硬性指标：若某个原子任务经历3次“实现-测试”循环仍未通过，或审查Agent发现高危漏洞，立即转交人类处理，不再使用Agent自动修复。
        
    *   这防止了定理5.1的恶性循环，即Agent在错误方向上不断迭代浪费认知资源。
        

#### 五、总结：一句话心法

Agent是你的工匠团队，每个工匠只会做自己那一道工序，而图纸（接口）、验收标准（测试）和最终印章（合入）始终在你手里。 让Agent在严密的契约和测试下运行，你就能同时获得速度与可控性。
