01 背景与问题 变与不变

📈 变了什么？

  • 编码效率提升 10× 以上
  • 反馈周期从小时级压缩到分钟级
  • AI 可辅助需求分析、测试、审查全流程

🔒 什么没变？

  • 软件工程的本质：将人类意图转化为健壮、可维护的逻辑系统
  • 核心挑战：复杂性、协作、变更管理
  • 终极责任：人类开发者对最终产品负责
核心洞察：AI 没有改变软件工程的底层逻辑，而是放大了某些规律。遵循第一性原理，才能真正驾驭 AI。
02 第一性原理 6 条底层逻辑
原理 1

意图清晰

混沌输入只会得到混沌输出。AI 能加速表达，但无法替代思考。

原理 2

所有权与可解释性

你交出去的是代码，不是责任。必须能解释、维护每一行关键代码。

原理 3

原子化与组合

复杂系统只能由简单、可验证的单元组合而来，不存在一次性生成的捷径。

原理 4

批判性验证

把 AI 的一切输出视为待证伪的假设。信任是逐步建立的，初始必须不信任。

原理 5

上下文即约束

AI 的“编程语言”是精确的上下文和约束。管理上下文就是管理 AI 的行为边界。

原理 6

认知循环加速

AI 缩减的是反馈周期，而不是思考的必要性。决策权与反思必须在人。

—— 这 6 条原理，是后续所有公理、定理和实战方案的根基
03 公理体系 5 条不可违背的工程公理
公理一 · 输入上限原理

AI 输出的质量上限，由输入意图清晰度、上下文完备性和约束精确性共同决定。

公理二 · 责任不可让渡

软件开发者对最终产品承担完全且不可转移的责任。AI 生成的代码必须可被人类完全解释、维护和调试。

公理三 · 原子组合必然性

可靠的复杂系统只能由独立可验证的原子单元确定性地组合而成，不存在从混沌需求直接生成可靠系统的一次性捷径。

公理四 · 输出可错性

AI 的每个输出在未经验证前，都仅仅是基于统计模式的假设，必然包含非零概率的错误、幻觉或安全缺陷。

公理五 · 认知循环守恒

软件开发的本质是“假设→验证→修正”的认知循环。AI 可压缩单次循环时间，但不可消除循环本身。

这 5 条公理经过事实性审查与逻辑重构，构成整套方法论的基石。
04 定理推导 13 条可操作工程定理

从公理一 · 输入上限

  • 定理 1.1 废进废出：输入歧义 → 输出必然存在对应不确定性
  • 定理 1.2 意图收益递减：超过 AI 能力边界后追加意图不再提高质量

从公理二 · 责任不可让渡

  • 定理 2.1 理解必然性：必须能独立解释每一行关键代码
  • 定理 2.2 签名即责任：合入代码视同开发者本人所写

从公理三 · 原子组合

  • 定理 3.1 一次性生成不可靠：正确性概率随代码长度指数下降
  • 定理 3.2 接口契约锁定：先定边界，后填内容
  • 定理 3.3 单元测试前置：没有测试的 AI 输出不具备可验证性

从公理四 · 输出可错性

  • 定理 4.1 验证不可替代：不存在跳过验证的捷径
  • 定理 4.2 信任与风险反比：风险越高，初始信任应越低
  • 定理 4.3 幻觉湮灭：只能通过外部事实源交叉验证根除

从公理五 · 认知循环守恒

  • 定理 5.1 循环不可跨越：每个增量必须经历至少一次完整循环
  • 定理 5.2 决策权定理：决策权不可委托给 AI
  • 定理 5.3 反思不可自动化：反思本身无法外包
总计：5 条公理 → 13 条定理，无逻辑矛盾。

05 实战方案 · 操作详解 关键步骤与检查点

阶段一：需求原子化

  • 强制拆分为“一个函数/类只做一件事”，填写“任务委托书”
  • 检查点：任务描述超一屏或含“和/或”等并列词 → 继续拆分

阶段二：接口契约锁定

  • 人类先写函数签名、类型定义，写入项目规则文件
  • 检查点：接口是否有版本号？异常路径是否已定义？

阶段三：TDD 循环生成

  • 先写测试用例 → AI 生成实现 → 运行测试 → 最多循环 3 次
  • 节奏：一个原子任务 = 一个 TDD 循环，理想耗时 5-15 分钟

阶段四：批判性验证（四层防御）

  • Layer 1：静态检查（TypeScript/ESLint 零错误）
  • Layer 2：动态检查（单元+集成全绿）
  • Layer 3：人工审查（高风险逐行）
  • Layer 4：外部事实交叉验证（官方文档校验）

阶段五：所有权确认

  • 橡皮鸭解释：向同事解释，不通则标记为债务
  • Git + ADR：禁止写“AI generated”，记录方案与限制
  • 杀死黑箱：不允许存在“能工作但不知道为什么”的模块
06 多 Agent 分工体系 契约驱动 · 人类决策
铁律：Agent 只是专业助手，分担执行；人类保留决策、验收和最终责任。
角色

👤 人类主工程师

定义目标、拆分任务、设计接口、最终验收、承担责任

辅助

📋 规划 Agent

辅助需求原子化，生成任务委托书草稿（人类审核）

核心

⚙️ 实现 Agent

在锁定接口和测试下生成代码，不得修改接口、不得逃避测试

核心

🧪 测试 Agent

编写测试用例、执行测试、报告结果

验证

🔍 审查 Agent

静态分析、模式检查、常见漏洞扫描

辅助

📄 文档 Agent

生成接口文档、更新 ADR 草稿

协作法则：所有 Agent 不直接对话，只通过人类定义的“契约”间接协作。 人类定义接口 + 测试 → 实现 Agent → 测试 Agent → 审查 Agent → 人类终审。
06 Agent 分工 · 流程植入 五阶段中的 Agent 协作
阶段一 · 需求原子化

规划 Agent 生成任务拆解建议 → 人类审核确认

阶段二 · 接口契约锁定

实现 Agent 生成接口草案 → 人类确认 → 文档 Agent 生成 OpenAPI 草稿

阶段三 · TDD 循环生成（串行）

测试 Agent 先写测试 → 人类审核 → 实现 Agent 编码 → 测试 Agent 验证 → 最多 3 次

阶段四 · 批判性验证（并行）

审查 Agent 静态扫描 + 测试 Agent 动态验证 + 依赖审计 → 人类聚焦高风险

阶段五 · 所有权确认

文档 Agent 生成 ADR 草稿 → 人类修正并签名合入

三大安全准则： ① 自主权阶梯（信任从零开始） ② 上下文隔离（只交换契约） ③ 紧急熔断（3 次失败转人工）
◆ 总结 · 核心心法 一句话操作指南
“把 AI 当成一个速度极快但需要你承担全部责任的初级程序员”
你怎么带新人，就怎么带 AI

📝 你的角色

  • 你写任务书（阶段一）
  • 你定接口（阶段二）
  • 你写测试（阶段三）
  • 你验收（阶段四）
  • 你签责任状（阶段五）

🎯 AI 的角色

  • 只填充“实现”这一个环节
  • 在严密契约和测试下运行
  • 整个流程的节奏和质量由你掌控
杠杆越长，支点必须越稳。 那个支点就是你的判断力、责任心和工程纪律。
